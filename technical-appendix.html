1. Introduction and purpose of the STEPS tool

The STEPS FETP Scale Up Decision Aid is an interactive decision support tool that combines preference evidence, cost templates and outbreak impact assumptions to help policy makers design and assess Field Epidemiology Training Programme (FETP) scale up scenarios in India.

The tool links three elements. First, a discrete choice experiment analysed using a mixed logit model provides predicted endorsement and an implied willingness to pay for different programme packages. Second, tier specific cost templates translate an assumed cost per trainee per month into programme costs, and optionally into broader economic costs when opportunity cost is included. Third, an epidemiological module converts expected graduates into expected outbreak responses and an indicative outbreak related monetary benefit over a specified planning horizon using a standard discounting approach.

STEPS is scenario based. Users choose a training tier, career incentive, mentorship intensity and delivery mode, then specify cost per trainee per month, trainees per cohort and the number of cohorts. The tool reports endorsement rates, willingness to pay measures, programme and economic costs, outbreak related epidemiological outputs, benefit cost ratios and net benefits. Outputs are intended for structured discussion and comparison of options, not as forecasts.

Important implementation note

In the current tool build, the outbreak response time attribute is fixed to “detect and respond within 7 days” in the user interface. The preference model and epidemiological module therefore both apply the “7 day” level for all scenarios.

2. Overview of the preference study and mixed logit model

2.1 Discrete choice experiment and attribute set

The preference study elicited stakeholder preferences using a discrete choice experiment in which respondents evaluated hypothetical FETP programme options compared with an opt out alternative representing no new FETP funding. Attributes reflected programme design choices that are relevant for scale up planning and that can be adjusted by ministries and partners.

Tier Frontline, Intermediate, Advanced.

Career incentive Certificate, University qualification, Government career pathway.

Mentorship Low, Medium, High.

Delivery Blended, Fully in person, Fully online.

Response time 30 days, 15 days, 7 days (in the study; fixed to 7 days in the current tool build).

Cost Cost per trainee per month (INR).

2.2 Mixed logit model used in the tool

The tool uses mean coefficients from a mixed logit preference model to compute (i) a utility index for a configured programme option, (ii) a utility index for the opt out alternative, and (iii) a predicted endorsement probability for the programme option relative to opt out. Mixed logit is widely used in discrete choice experiments because it can accommodate preference heterogeneity across respondents rather than imposing a single set of preferences for all decision makers (Train, 2009; Hensher, Rose and Greene, 2015).

2.3 Coefficients implemented in the tool

The implemented coefficients are fixed values in the application script and are applied consistently for all scenarios. Level zero indicates the reference category for each attribute.

4. Cost construction methodology

STEPS uses a cost per trainee per month input and translates it into programme costs per cohort and, optionally, total economic costs per cohort by applying tier specific opportunity cost rates.

4.1 Programme cost concept

The programme cost per cohort is a direct financial cost concept constructed from the user input “cost per trainee per month”, multiplied by the tier duration and by the number of trainees in the cohort. This direct cost is the base for cost breakdown tables and for the opportunity cost calculation.

4.2 Economic cost concept and opportunity cost

The tool also supports an economic cost concept that adds an opportunity cost component. In STEPS, the opportunity cost is implemented as a tier specific rate applied to the programme cost per cohort. When the opportunity cost toggle is off, the opportunity cost rate is set to zero and the total economic cost equals the programme cost.

Opportunity cost in the tool

Opportunity cost is modelled as a proportional add on to the programme cost rather than as a separate per trainee salary input. This is intentional in the current build because it ensures a consistent economic cost scaling structure across tiers using the combined templates.

5. Cost component shares and scaling logic

Each tier has a combined cost template that includes a list of direct cost components (with shares that sum to 1 across the listed items) and an opportunity cost rate. In the Costing tab, STEPS reports the monetary amount for each component by multiplying the programme cost per cohort by the component share. It also reports the implied component cost per trainee per month by dividing the component amount by (trainees per cohort × tier months).

6. Programme cost per trainee and cohort calculations

6.1 Programme cost per cohort

The tool treats the user input “cost per trainee per month” as a direct monthly programme cost. It converts this to a programme cost per cohort by multiplying by the tier duration (months) and the number of trainees per cohort.

Programme cost per cohort (as implemented)
programmeCostPerCohort = costPerTraineePerMonth × tierMonths × traineesPerCohort
This value is reported in the Results tab as “Programme cost per cohort” and in the Costing tab summary cards.

6.2 Component amounts and per trainee per month display

If a tier template is available, each component amount is calculated by applying the component share to the programme cost per cohort. The tool then divides by (trainees per cohort × tier months) to show an implied component cost per trainee per month. These displays are for transparency about what the combined template represents.

Component display calculations (Costing tab)
componentAmount = programmeCostPerCohort × directShare componentPerTraineePerMonth = componentAmount ÷ (traineesPerCohort × tierMonths)
These calculations do not change total costs. They provide a consistent breakdown for reporting and discussion.

7. Opportunity cost inclusion and exclusion logic

7.1 Opportunity cost per cohort

When the opportunity cost toggle is on, STEPS applies the tier specific opportunity cost rate from the combined template to the programme cost per cohort. When the toggle is off, the applied opportunity cost rate is set to zero.

Opportunity cost and total economic cost per cohort (as implemented)
if opportunityCostIncluded: oppRate = tierTemplateOppRate else: oppRate = 0 opportunityCostPerCohort = programmeCostPerCohort × oppRate totalEconomicCostPerCohort = programmeCostPerCohort + opportunityCostPerCohort
Total economic cost per cohort is the cost denominator for the benefit cost ratio and net benefit outputs shown in the Results tab.

7.2 Interpretation of “economic cost” in STEPS

Economic cost in STEPS is a practical concept intended to broaden the resource base when opportunity cost is included. It aggregates programme costs and an additional opportunity cost component using a tier specific proportional rule. This is appropriate for comparing scenarios consistently within the tool and for sensitivity checks. It should not be interpreted as an audited expenditure figure.

8. Willingness to pay calculations

8.1 Utility index and endorsement probability

For a configured programme option, the tool forms a utility index by summing the programme constant, the relevant attribute coefficients for the selected levels, the response time coefficient (fixed to 7 days in the current build), and the cost term. The cost term uses cost per trainee per month expressed in thousands of INR.

Utility index (programme option)
costThousands = costPerTraineePerMonth ÷ 1000 U_programme = ASC_programme + tierEffect + careerEffect + mentorshipEffect + deliveryEffect + responseEffect + costCoefficient × costThousands
The opt out alternative has a fixed utility index equal to ASC_optout.

Endorsement is then calculated using a two option logit share between the programme and opt out. STEPS reports endorsement and opt out rates as percentages.

Endorsement probability (two option logit share)
endorseProb = exp(U_programme) ÷ (exp(U_programme) + exp(U_optout)) endorseRatePercent = 100 × endorseProb optOutRatePercent = 100 − endorseRatePercent
In the tool, endorsement is the predicted share of stakeholders choosing the configured programme option rather than the opt out alternative.

8.2 WTP per trainee per month (INR)

STEPS computes willingness to pay per trainee per month by taking the non cost component of the programme utility and dividing by the absolute value of the cost coefficient. Because the cost coefficient is defined per 1,000 INR, the ratio is scaled by 1,000 to express willingness to pay in INR per trainee per month.

WTP per trainee per month (as implemented)
nonCostUtility = ASC_programme + tierEffect + careerEffect + mentorshipEffect + deliveryEffect + responseEffect WTP_perTraineePerMonth_INR = (nonCostUtility ÷ |costCoefficient|) × 1000
This value is an implied measure from stated preferences. It is not a budget or a market price.

8.3 Total WTP per cohort and across cohorts

The tool scales WTP per trainee per month to the cohort level using tier duration (months) and trainees per cohort. It then aggregates across the number of cohorts.

WTP scaling (as implemented)
WTP_perCohort = WTP_perTraineePerMonth × tierMonths × traineesPerCohort WTP_allCohorts = WTP_perCohort × numberOfCohorts
By default, WTP aggregation does not automatically apply the endorsement rate. Endorsement adjusted WTP is available as a sensitivity definition in the sensitivity tables.

8.4 WTP outbreak component in the sensitivity summary

The sensitivity summary also reports an “outbreak component” of WTP as a fixed 30 percent share of total WTP across all cohorts. This is an internal reporting convention used in the tool to provide a simple decomposition for discussion.

WTP outbreak component (as implemented)
WTP_outbreakComponent_allCohorts = 0.3 × WTP_allCohorts

9. Epidemiological outputs and outbreak benefit construction

The epidemiological module maps trained graduates into outbreak responses per year and then into a discounted outbreak related monetary benefit over the planning horizon.

9.1 Default epidemiological settings

STEPS initialises a set of epidemiological assumptions that apply to all tiers unless changed in the Settings tab. Defaults include a planning horizon of 5 years, a discount rate of 3 percent per year, a completion rate of 90 percent, outbreaks handled per graduate per year of 0.5, and a value per outbreak of INR 4,000,000.

9.2 Present value factor

STEPS discounts a constant annual outbreak benefit stream over the planning horizon using an annuity style present value factor. If the discount rate is zero, the factor equals the number of years. If the discount rate is positive, the factor is the standard discounted sum of 1 unit per year.

Present value factor used in the tool
PV_factor = (1 − (1 + r)^(-T)) ÷ r
r is the annual discount rate and T is the planning horizon in years.

9.3 Graduates produced per cohort and across cohorts

For each cohort, the tool first applies completion to enrolled trainees, then applies an endorsement adjustment. The endorsement adjustment uses the endorsement rate from the preference model and is implemented as a proportional factor. This means that epidemiological outputs represent an endorsement weighted effective scale.

Graduates (as implemented)
completedPerCohort = traineesPerCohort × completionRate graduatesEffectivePerCohort = completedPerCohort × (endorseRatePercent ÷ 100) graduatesAllCohorts = graduatesEffectivePerCohort × numberOfCohorts

9.4 Outbreak responses per year

Outbreak responses per year are calculated from effective graduates, outbreaks per graduate per year, and the response time multiplier. Because response time is fixed to 7 days in the current build, the multiplier is always 1.5.

Outbreak responses per year (as implemented)
outbreaksPerYearPerCohort = graduatesEffectivePerCohort × outbreaksPerGraduatePerYear × responseMultiplier outbreaksPerYearNational = outbreaksPerYearPerCohort × numberOfCohorts

9.5 Outbreak benefit per cohort (discounted)

The annual outbreak benefit per cohort is calculated as outbreak responses per year per cohort multiplied by value per outbreak. The present value over the planning horizon is then obtained by multiplying by the present value factor.

Outbreak benefit per cohort (as implemented)
annualOutbreakBenefitPerCohort = outbreaksPerYearPerCohort × valuePerOutbreak outbreakPVPerCohort = annualOutbreakBenefitPerCohort × PV_factor epiBenefitPerCohort = outbreakPVPerCohort (graduate benefit currently set to 0)

Interpretation: This is an indicative outbreak related monetary benefit driven by assumptions and the endorsement weighted scale. It is not an observed saving and it should be treated as a structured scenario metric.

10. Value per outbreak presets and sensitivity inputs

10.1 Preset values in the sensitivity dropdowns

The sensitivity dropdowns include presets expressed in millions of rupees: 4, 5, 10, 20, 30 and 40. Selecting a preset updates the value per outbreak for all tiers and recalculates the current scenario outputs.

10.2 Parsing user provided values

The tool supports flexible entry formats for outbreak values in settings fields. It recognises “million” and “mn”, “billion” and “bn”, and “crore” and “cr”. If a numeric value is entered without a unit and it is below 1000, STEPS interprets it as millions of rupees. For example, “20” is treated as 20 million INR.

Examples of outbreak value interpretation in STEPS

Input “4” is interpreted as 4,000,000 INR.

Input “2 crore” is interpreted as 20,000,000 INR.

Input “0.5 bn” is interpreted as 500,000,000 INR.

Input “4000000” is interpreted as 4,000,000 INR.

11. Benefit cost ratios and net benefits

11.1 Cohort level benefit cost ratio and net benefit

STEPS reports an epidemiological benefit cost ratio per cohort as the ratio of discounted outbreak benefit per cohort to total economic cost per cohort. It also reports net benefit per cohort as benefit minus total economic cost.

Cohort level BCR and net benefit (as implemented)
BCR_perCohort = epiBenefitPerCohort ÷ totalEconomicCostPerCohort netBenefitPerCohort = epiBenefitPerCohort − totalEconomicCostPerCohort

11.2 National level benefit cost ratio and net benefit

National totals are computed by multiplying cohort level values by the number of cohorts. The national benefit cost ratio is the ratio of total outbreak benefit across cohorts to total economic cost across cohorts. Net benefit across cohorts is total benefit minus total cost.

National aggregation (as implemented)
totalEconomicCost_allCohorts = totalEconomicCostPerCohort × numberOfCohorts epiBenefit_allCohorts = epiBenefitPerCohort × numberOfCohorts BCR_national = epiBenefit_allCohorts ÷ totalEconomicCost_allCohorts netBenefit_allCohorts = epiBenefit_allCohorts − totalEconomicCost_allCohorts

Note: In the Results and National tabs, the benefit cost ratios shown are based on outbreak related epidemiological benefits only, not on willingness to pay.

12. National aggregation across cohorts

STEPS provides a National Simulation tab that summarises the configured scenario at the scale of all cohorts. It reports total economic cost, total outbreak related benefit, national benefit cost ratio, total graduates over the timeframe and outbreak responses per year at national scale.

The national graduates output aggregates endorsement weighted graduates across cohorts. The national outbreak responses per year output aggregates cohort level outbreak responses per year across cohorts. Because the response time is fixed to 7 days, response multipliers are constant across scenarios.

How endorsement affects national epidemiological outputs

In STEPS, endorsement affects epidemiological outputs directly because graduates are multiplied by endorsement share when computing effective graduates. This links preference evidence to the implied effective scale of implementation.

13. Sensitivity analysis tables and benefit definitions

13.1 Sensitivity tables produced by STEPS

STEPS produces two related sensitivity tables. The first table is a headline summary that mixes costs, epidemiological totals and willingness to pay metrics for the current configuration and any saved scenarios. The second table is a more detailed view that reports cohort level values alongside model and scenario descriptors.

13.2 Outbreak benefit inclusion toggle

In the sensitivity headline table, users can switch outbreak benefits on or off. When outbreak benefits are excluded, combined benefits in that table are set equal to total willingness to pay, and the combined benefit cost ratio is calculated using willingness to pay as the benefit numerator.

Interpretation of the sensitivity benefit cost ratios

The sensitivity headline table can report two benefit cost ratios: a DCE only ratio using total willingness to pay divided by total economic cost, and a combined ratio that adds outbreak benefits to willingness to pay when outbreak benefits are included.

13.3 Endorsement adjusted benefit definition

The sensitivity headline table supports an endorsement adjusted definition of willingness to pay. Under this definition, total willingness to pay is multiplied by a chosen endorsement rate. Users can either use each scenario’s predicted endorsement rate or override it with a single endorsement rate applied to all scenarios.

Endorsement adjusted WTP in sensitivity mode (as implemented)
effectiveWTP_allCohorts = WTP_allCohorts × (endorsementUsed ÷ 100)

13.4 Net benefit columns in the headline table

The headline table includes a “Net benefit (mn INR)” column that reflects outbreak related net benefit (outbreak benefits minus economic costs) because it is computed from the epidemiological totals. Additional net present value columns report DCE only net value (WTP minus cost) and combined net value (WTP plus outbreak benefit minus cost, subject to whether outbreak benefits are included).

14. Worked examples aligned to the STEPS calculations

This section provides worked examples that follow the same steps and parameter values implemented in the tool. Values are illustrative and are designed to help audit the calculations.

Example A. Intermediate tier, medium mentorship, blended delivery, university qualification

Scenario inputs

Step 1. Utility and endorsement

Tier effect (Intermediate) = 0.220.

Career effect (University qualification) = 0.017.

Mentorship effect (Medium) = 0.453.

Delivery effect (Blended) = 0.000.

Response effect (7 days) = 0.610.

Programme constant = 0.168 and opt out constant = -0.601.

Cost thousands = 50,000 ÷ 1,000 = 50, so cost term = -0.005 × 50 = -0.25.

U programme = 0.168 + 0.220 + 0.017 + 0.453 + 0.000 + 0.610 - 0.250 = 1.218.

U opt out = -0.601.

Endorsement probability = exp(1.218) ÷ (exp(1.218) + exp(-0.601)) ≈ 0.8604, or 86.04 percent.

Opt out rate ≈ 13.96 percent.

Step 2. WTP per trainee per month and total WTP

Non cost utility = 0.168 + 0.220 + 0.017 + 0.453 + 0.000 + 0.610 = 1.468.

WTP per trainee per month = (1.468 ÷ 0.005) × 1,000 = 293,600 INR.

WTP per cohort = 293,600 × 12 × 40 = 140,928,000 INR.

WTP across all cohorts = 140,928,000 × 3 = 422,784,000 INR.

WTP outbreak component (reporting convention) = 0.3 × 422,784,000 = 126,835,200 INR.

Step 3. Programme cost, opportunity cost, economic cost

Programme cost per cohort = 50,000 × 12 × 40 = 24,000,000 INR.

Intermediate opportunity cost rate = 0.35, so opportunity cost per cohort = 24,000,000 × 0.35 = 8,400,000 INR.

Total economic cost per cohort = 24,000,000 + 8,400,000 = 32,400,000 INR.

Total economic cost across all cohorts = 32,400,000 × 3 = 97,200,000 INR.

Step 4. Graduates, outbreak responses and outbreak benefit

Completed per cohort = 40 × 0.90 = 36.

Endorsement factor = 86.04 percent, so effective graduates per cohort = 36 × 0.8604 ≈ 30.974.

Total effective graduates across cohorts = 30.974 × 3 ≈ 92.922.

Response multiplier (7 days) = 1.5. Outbreak responses per year per cohort = 30.974 × 0.5 × 1.5 ≈ 23.231.

Outbreak responses per year national = 23.231 × 3 ≈ 69.694.

Present value factor for 5 years at 3 percent is (1 − (1.03)^(-5)) ÷ 0.03 ≈ 4.5797.

Annual outbreak benefit per cohort = 23.231 × 4,000,000 ≈ 92,925,000 INR.

Discounted outbreak benefit per cohort = 92,925,000 × 4.5797 ≈ 425,548,000 INR.

Discounted outbreak benefit across all cohorts ≈ 1,276,644,000 INR.

Step 5. Benefit cost ratios and net benefits (outbreak benefits only)

Benefit cost ratio per cohort = 425,548,000 ÷ 32,400,000 ≈ 13.13.

Net benefit per cohort = 425,548,000 − 32,400,000 ≈ 393,148,000 INR.

National benefit cost ratio = 1,276,644,000 ÷ 97,200,000 ≈ 13.13.

National net benefit ≈ 1,276,644,000 − 97,200,000 ≈ 1,179,444,000 INR.

How to use the worked example

If you enter the same inputs into STEPS (tier, attributes, cost, trainees, cohorts, and keep default epidemiological settings), the endorsement rate, WTP measures, costs, graduates, outbreaks, outbreak benefit, benefit cost ratios and net benefits should align closely with the values above, allowing for minor differences from rounding in display formatting.

15. Copilot interpretation and JSON export structure

15.1 Purpose of the Copilot tab

The Copilot tab prepares a narrative briefing prompt that combines a structured set of instructions with a JSON export of the current scenario. The prompt instructs Copilot to write a narrative policy brief using only the scenario JSON as quantitative evidence, and to report values primarily in Indian rupees.

15.2 JSON structure exported by STEPS

The JSON export includes metadata, configuration values, preference model outputs (endorsement and WTP), cost results, epidemiological results, and benefit cost outputs. The structure below summarises the major blocks.

metadata: generatedAt, tool, country, planningHorizonYears, discountRateForBenefits configuration: tier, careerIncentive, mentorshipIntensity, deliveryMode, responseTimeDays, costPerTraineePerMonthINR, traineesPerCohort, cohorts, opportunityCostIncluded preferenceModel: endorsementRatePercent, optOutRatePercent, wtpPerTraineePerMonthINR, wtpPerCohortINR, wtpAllCohortsINR costResults: programmeCostPerCohortINR, opportunityCostPerCohortINR, economicCostPerCohortINR, totalEconomicCostAllCohortsINR epidemiologicalResults: graduatesPerCohort, graduatesAllCohorts, outbreakResponsesPerYearPerCohort, outbreakResponsesPerYearNational, epiBenefitPerCohortINR, epiBenefitAllCohortsINR benefitCostResults: bcrPerCohortEpidemiological, bcrNationalEpidemiological, netBenefitPerCohortINR, netBenefitAllCohortsINR, wtpOutbreakComponentAllCohortsINR, totalWtpAllCohortsINR

15.3 Interpretation guardrails

The Copilot instructions explicitly request that Copilot treats all numbers in the JSON as internally consistent and uses them as the quantitative evidence base. This ensures the narrative briefing remains aligned to the tool outputs and avoids introducing external numeric assumptions during drafting.

16. Summary of assumptions, limitations and references

16.1 Summary of key assumptions

The tool relies on assumptions that make scenario comparisons transparent and reproducible. Key assumptions include fixed tier durations (3, 12, 24 months), a fixed outbreak response time of 7 days in the current build, and a cost construction approach that treats cost per trainee per month as the base programme cost. Epidemiological outputs depend on settings for completion, outbreaks per graduate per year, value per outbreak, planning horizon and discount rate. WTP is computed from preference coefficients using a marginal rate of substitution with the cost coefficient.

16.2 Practical limitations

First, willingness to pay values are implied from stated preferences and are intended as a structured measure of relative value, not as market prices or budgets. Second, outbreak benefits are indicative and depend on the assumed value per outbreak and on simple mappings from graduates to outbreak responses. Third, the opportunity cost module is implemented as a proportional add on rather than a direct wage based estimate, which supports standardised comparisons but may differ from alternative accounting approaches. Fourth, because response time is fixed to 7 days in the current build, STEPS does not currently support comparing response time levels within the interface.
